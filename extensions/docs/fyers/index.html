Fyers Stop Loss Manager

🛡️

Cloudflare/WAF Resilience & Fallback Design

Because the Fyers backend API restricts programmatic fetches from cross-origin contexts (returning 503 Service Unavailable or block errors), this extension employs a hybrid execution mode. When the Background Service Worker is rate-limited or blocked, the controller silently falls back to dynamic DOM scraping. The scraped data is then mapped to the intercepted order book to guarantee uninterrupted average price and stop-loss metrics.

🔌 Page Context

Runs directly inside the tab's JS runtime context, giving it access to hook page-level fetch methods.

  • Script: fyers-bridge.js
  • Hooks window.fetch to read headers.
  • Scans cookies & storage for JWT tokens.
  • Dispatches token credentials to Content Script.

🧠 Content Script

Main supervisor running in Chrome's isolated sandbox. Manages local UI rendering and loops.

  • Script: content.js
  • Renders floating panel and table list.
  • Performs timeline polling triggers.
  • Initiates fallback DOM scraping loops.
  • Coordinates speech synthesis alerts.

⚙️ Service Worker

Chrome Extension background script. Runs with privileged permissions to bypass CORS rules.

  • Script: background.js
  • Listens for fetch requests from Content Script.
  • Calls Fyers endpoints via background fetch.
  • Performs backup retry logic on public APIs.
  • Bypasses standard tab CORS constraints.

Detailed Step Execution Timeline

Token Capture Phase Step 1

On initial page load, fyers-bridge.js is injected and hooks window.fetch. It scans localStorage, sessionStorage, and cookies for Bearer credentials. Once found (or when intercepted from network calls), it fires a custom event:

window.dispatchEvent(new CustomEvent("fyers-token-captured", { detail: { token: "Bearer eyJ...", baseUrl: "https://api-y1.fyers.in" } }));

Data Fetch & Background Routing Step 2

The content script receives the credentials and routes them to background.js via chrome.runtime.sendMessage. The background worker attempts to query Fyers API paths:

// GET Fyers Web Portal API v3 Paths: - https://api-t1.fyers.in/trade/v3/positions/v2 - https://api-t1.fyers.in/trade/v3/orders/v2

Dynamic DOM Scraper & Fallback Mapping Step 3

If the API call fails, the scraper parses the DOM table. It maps the column headers dynamically by reading header texts (e.g. th text containing "Symbol", "Avg. Price", etc.). Next, it matches the scraped positions to the active orders book using symbol normalization:

// Strips exchange prefixes ("NSE:") and suffixes ("-EQ") for accurate lookup function normalizeSymbol(sym) { return sym.replace(/^(NSE:|BSE:|MCX:|NFO:)/i, "").replace(/-EQ$/i, "").trim().toUpperCase(); }

SL Trailing & Modifier Execution Step 4

When the user clicks the ↑ (Move SL) button, the content script checks if the order has an active slOrderId. If yes, it requests the background worker to dispatch a CORS-free modification patch to Fyers. If no API order ID is available, it automates filling Fyers' native Protect Modal:

// Background PATCH for modifying Stop Loss on bracket orders (BO) PATCH https://api-t1.fyers.in/trade/v3/orders/bo Body: { "id": orderId, "stopLoss": newSL, "qty": qty }
